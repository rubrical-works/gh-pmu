## Findings

### Medium

#### M1 — Raw GraphQL query builders use fragile `%q` interpolation
- **Files:** `internal/api/mutations.go:843`; `internal/api/queries.go:905, 939, 1340, 1416, 1643, 1729, 2376, 2549`
- **Category:** security / api-design
- **Severity:** medium
- **Description:** Query strings are built with `fmt.Sprintf` using Go `%q` to interpolate owner, repo, label names, and node IDs. Go and GraphQL string-literal escape rules overlap but differ on edge cases.
- **Recommendation:** Use variable-based queries (`doRawGraphQLBody` with a `variables` map) or add a `validateGraphQLIdent` helper akin to `validateGitRef` (`mutations.go:1347`) applied at each call site.
- **Issue:** #832

#### M2 — `intake --apply` issues N×3 API calls
- **Files:** `cmd/intake.go:205-257`; anti-pattern described in `internal/api/mutations.go:155-168`
- **Category:** performance
- **Severity:** medium
- **Description:** For N untracked issues, intake calls `AddIssueToProject` once and `SetProjectItemField` up to twice per issue; the latter internally re-fetches project fields every call. A 50-issue backfill runs ≥150 unnecessary schema fetches.
- **Recommendation:** Pre-fetch project fields once, switch to `SetProjectItemFieldWithFields`. Extend the `intakeClient` interface accordingly.
- **Issue:** #833

#### M3 — `config verify` silently discards checksum errors
- **Files:** `cmd/config.go:96-97`
- **Category:** observability
- **Severity:** medium
- **Description:** Errors from `integrity.ComputeChecksum` and `integrity.LoadChecksum` are discarded with `_`. Users can't distinguish "no checksum file yet" from "I/O error".
- **Recommendation:** Surface the error via `cmd.ErrOrStderr()` and document that `LoadChecksum` returns `("", nil)` when absent.
- **Issue:** #834 (bundled with L7)

#### M4 — Owner/repo parameters unvalidated before interpolation
- **Files:** `internal/api/queries.go:1625`; related mutation/query entry points
- **Category:** security / api-design
- **Severity:** medium
- **Description:** `fmt.Sprintf("repo:%s/%s", owner, repo)` and similar sites interpolate owner/repo with no character-set enforcement. Today's callers come from `.gh-pmu.json` (trusted), but the absence of a validator at the API boundary is a latent injection surface.
- **Recommendation:** Add `validateOwner` / `validateRepo` helpers (modeled on `validateGitRef`) and invoke at the top of query/mutation entry points.
- **Issue:** #835
